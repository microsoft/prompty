The Prompty Runtime Specification defines the runtime behavior of Prompty ‚Äî a markdown file format and execution pipeline for LLM prompts. It is language-agnostic: implementations in any programming language MUST conform to the behavioral contracts defined here.

This specification is intended for **implementers building Prompty runtimes** in any language. It defines the behavioral contracts that all implementations must satisfy. For user-facing guides on using Prompty, see the [Getting Started](/getting-started/) section.

**Version**: 0.1.0-draft ¬∑ **Status**: Draft

:::note
The type system (data model) is defined in TypeSpec under `schema/` and emitted
into the runtime libraries and [Schema Reference](/reference/). This
specification is the source of truth for runtime behavior; it covers what
implementations **do** with those generated types.
:::

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt).

## Public API Surface

Implementations MUST expose the following functions. Each function MUST have an async variant (e.g., `load_async`). Each function SHOULD have a sync variant.

| Function | Signature | Returns | Spec Section |
|----------|-----------|---------|-------------|
| `load` | `(path: string) ‚Üí PromptAgent` | PromptAgent | [¬ß4](/specification/loading/) |
| `render` | `(agent, inputs) ‚Üí string` | Rendered string | [¬ß5](/specification/rendering/) |
| `parse` | `(agent, rendered) ‚Üí Message[]` | Message list | [¬ß6](/specification/parsing/) |
| `prepare` | `(agent, inputs) ‚Üí Message[]` | Message list | [¬ß5](/specification/rendering/) + [¬ß6](/specification/parsing/) |
| `run` | `(agent, messages) ‚Üí result` | Processed result | [¬ß7](/specification/wire-format/) + [¬ß8](/specification/processing/) |
| `invoke` | `(path, inputs) ‚Üí result` | Processed result | [¬ß4](/specification/loading/)‚Äì[¬ß8](/specification/processing/) |
| `turn` | `(path, inputs, tools?) ‚Üí result` | Processed result | [¬ß4](/specification/loading/)‚Äì[¬ß8](/specification/processing/) + loop |
| `process` | `(agent, response) ‚Üí result` | Extracted result | [¬ß8](/specification/processing/) |
| `validate_inputs` | `(agent, inputs) ‚Üí validated_inputs` | Validated dict | [¬ß12](/specification/cross-cutting/) |

**Async contract**: All pipeline functions MUST have async variants. Implementations SHOULD also provide sync variants. When both exist, they MUST produce identical results for the same inputs.

## Conformance Levels

This specification uses three conformance levels:

- **MUST** ‚Äî An absolute requirement. An implementation that does not satisfy a MUST requirement is non-conformant.
- **SHOULD** ‚Äî Recommended but not required. An implementation may omit a SHOULD requirement with good reason, but the implications must be understood.
- **MAY** ‚Äî Truly optional. An implementation may provide or omit a MAY feature freely.
